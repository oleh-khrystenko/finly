# Sprint 16 — Public Help Docs + AI Chat — Planning Questions

Гібридна архітектура: один документ як джерело правди, що (1) рендериться як публічна SEO-сторінка довідки і (2) є базою знань для публічного AI-чату поверх неї.

## Питання (статус)

### Q1. Розміщення і URL
Де живе сторінка довідки і на якому host?
Гіпотеза: публічний anon-роут на кабінет-host (`finly.com.ua/help` або `/docs`), доступний без логіну, індексований. Не на `pay.` host (той суто для платіжних сторінок).

**Відповідь:** `finly.com.ua/help`, публічний anon-роут на основному host.

### Q2. Структура контенту
Одна довга сторінка-довідка vs багаторозділовий help-center з навігацією (кілька статей, sidebar/secції, окремі URL)?
Це впливає на SEO (окремі статті ранжуються краще), на модель контенту і на навігацію.

**Відповідь:** Багаторозділовий help-center. `/help` як індекс + окремі статті за власними URL, навігація. AI відсилає на конкретний розділ.

### Q3. Хто і як редагує документацію (джерело правди + ownership)
Документація — markdown/MDX у репозиторії, що його редагує розробник у PR? Чи потрібна можливість редагувати не-розробнику (CMS/адмінка)?
Окремо: API (для grounding AI) і Web (для рендеру) — два різні workspace. Хто канонічний власник тексту і як обидва його споживають з одного джерела, щоб не було drift.

**Відповідь:** Markdown-файли в репозиторії як єдине джерело правди, редагує розробник у PR. Жодної другої копії: Web рендерить ті самі файли у сторінки, API підтягує той самий текст у системний промпт AI. CMS/адмінка — поза скоупом (over-engineering на цьому етапі).

### Q4. Скоуп відповідей AI і guardrails
AI відповідає строго на "як користуватись Finly" і відмовляється від загальних питань (податкові/бухгалтерські/юридичні поради, "як рахувати ЄСВ" тощо)?
Для фінансового продукту відповідь на податкове питання — це ризик відповідальності. Гіпотеза: жорстко скоупимо на продукт, на off-topic ввічливо відмовляємось і відсилаємо до фахівця.

**Відповідь:** Жорсткий скоуп. AI відповідає тільки на "як працює і як користуватись Finly". Податкові/бухгалтерські/юридичні питання, як і будь-який off-topic, ввічливо відхиляє і відсилає до бухгалтера/фахівця. Не перетворюється на безкоштовний ChatGPT.

### Q5. Збереження anon-розмов
Чат для аноніма: ефемерний (нічого не пишемо в БД, історія тримається на клієнті в межах сесії) vs зберігаємо для аналітики/покращення?
Гіпотеза: ефемерний (приватність + простота), без прив'язки до userId. Аналітику (лічильники/теми) — окремо і пізніше, якщо треба.

**Відповідь:** Ефемерний. Нічого не пишемо в БД, історія живе на клієнті в межах сесії, без прив'язки до користувача. Аналітика — поза скоупом.

### Q6. Поведінка при вичерпанні денного бюджету
Коли спрацював глобальний денний circuit-breaker (захист гаманця): чат тимчасово недоступний з повідомленням "скористайтесь довідкою нижче", а статична сторінка завжди працює?
Гіпотеза: так, graceful degradation, статика ніколи не залежить від AI.

**Відповідь:** Так. При вичерпанні денного бюджету чат тимчасово недоступний з повідомленням-відсиланням на довідку, статична сторінка працює завжди й не залежить від AI.

### Q7. Воронка / CTA
Анонім читає довідку до реєстрації. Чи штовхаємо м'який CTA на реєстрацію (в сторінці і/або в чаті)?
Гіпотеза: так, ненав'язливо (банер/кнопка), чат може запропонувати "почати безкоштовно" коли доречно.

**Відповідь:** Так, м'який CTA. Ненав'язлива кнопка/банер на сторінці, чат може доречно запропонувати спробувати безкоштовно. Без агресії.
