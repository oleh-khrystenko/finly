# Sprint 16 — Public Help Center + Grounded AI Chat

## Мета і контекст

Дати Finly публічну довідку за адресою `finly.com.ua/help`: багаторозділовий help-center для ФОП і бухгалтерів, що пояснює, як працює сервіс. Поверх статичних статей живе публічний AI-чат, заземлений на той самий контент, який доступний анонімам без логіну.

Архітектурний стрижень: один набір markdown-статей у репозиторії як єдине джерело правди. Web рендерить їх у SEO-сторінки, API згодовує той самий текст AI як базу знань. Оновлення тексту в одному місці змінює і сторінку, і відповіді AI, тому документація не дрейфує від продукту.

Це водночас довідка і верхня частина воронки: анонім розуміє продукт ще до реєстрації, а інтерактивний AI демонструє ключову цінність Finly.

## Скоуп

- [ ] Набір markdown-статей довідки в репозиторії як єдине джерело правди (контент + метадані розділу)
- [ ] Публічний рендер: індекс `/help` + окремі статті за власними URL, навігація між розділами
- [ ] SEO для статичної частини (метатеги, canonical, sitemap, серверний рендер без залежності від AI)
- [ ] Публічний AI-чат на сторінці довідки: окремий unauthed endpoint, без списання executions
- [ ] Grounding AI строго на базу знань довідки з anti-hallucination поведінкою
- [ ] Жорсткий скоуп відповідей AI: тільки продукт, відмова від податкових/юридичних/off-topic питань
- [ ] Захист гаманця: IP-rate-limit для аноніма + глобальний денний бюджет-circuit-breaker
- [ ] Graceful degradation: при вичерпанні бюджету чат недоступний, статика працює завжди
- [ ] М'який CTA на реєстрацію (сторінка + доречні підказки чату)
- [ ] Переюз існуючого SSE-стріму і чат-UI з кабінету для anon-варіанту
- [ ] Запис у `docs/manual-checks` про перевірку anon-доступу, degradation і scope-guardrails

## НЕ-скоуп

- CMS чи адмінка для редагування довідки не-розробником. Контент правиться у PR. Якщо колись знадобиться контент-редактор без деплою, це окремий спринт.
- Зберігання anon-розмов у БД і аналітика тем. Чат ефемерний.
- RAG/векторний пошук по базі знань. Корпус малий, заземлення робиться повним інжектом у контекст.
- Мультимовність довідки. Single-locale uk, як і весь продукт.
- Збереження історії чату між сесіями для аноніма.

## Архітектурні рішення

### Єдине джерело правди для контенту
Статті — markdown-файли в репозиторії зі структурованими метаданими (заголовок, slug розділу, опис, порядок). Канонічний власник тексту один. Web будує з них сторінки, API будує з них базу знань для AI. Заборонена будь-яка друга копія тексту: і рендер, і grounding читають з одного набору файлів. Це механічно унеможливлює drift між тим, що користувач бачить очима, і тим, що переказує AI.

### Заземлення AI повним інжектом, без RAG
Корпус довідки малий і вміщається в контекст моделі з запасом. AI отримує всю базу знань у системному промпті, тому окрема retrieval-інфраструктура не потрібна. Системний промпт використовує prompt-caching (механізм уже застосований у поточному AI-провайдері), тому повторний інжект бази знань дешевий. AI інструктований відповідати тільки з наданого контенту, відсилати на конкретний розділ довідки і чесно визнавати, коли відповіді в базі немає.

### Публічний AI-endpoint окремо від кабінетного
Кабінетний `/ai/chat` лишається як є: auth, списання executions, збереження історії per-user. Anon-варіант — окремий публічний endpoint без auth, без executions-резервації, без запису в БД. Спільне між ними — лише низькорівневий SSE-стрім до провайдера. Anon-чат stateless на боці сервера: багатоходова розмова тримається на клієнті і дослається у запиті.

### Двошаровий захист гаманця
Перший шар, per-IP rate-limit для аноніма (на базі вже існуючого IP-лічильника AI-модуля, але зі своїм, нижчим лімітом і своїм namespace). Другий шар, глобальний денний бюджет-circuit-breaker: коли сумарна кількість anon-запитів за добу перевищує поріг, endpoint тимчасово відмовляє всім. Це стеля проти масового зловживання, якої немає у кабінетному варіанті, бо там витрати обмежені балансом користувача.

### Статика незалежна від AI
Сторінки довідки рендеряться на сервері з markdown без жодного звернення до AI. Чат, його rate-limit, його бюджет, доступність провайдера, жоден з цих факторів не впливає на можливість прочитати документацію. AI-чат, це прогресивне покращення поверх самодостатньої статичної довідки.

### Жорсткий scope-guardrail
Системний промпт явно обмежує AI темою продукту. Податкові, бухгалтерські, юридичні питання і будь-який off-topic отримують ввічливу відмову з відсиланням до фахівця. Це і захист від відповідальності за чужі фінансові рішення, і захист від перетворення чату на безкоштовний асистент загального призначення.

## Модель даних

Нових колекцій у БД немає. Anon-чат нічого не персистить.

База знань, це файлова структура в репозиторії, не сутність БД. Кожна стаття несе метадані: людський заголовок, URL-slug, короткий опис для SEO і для навігації, порядок у списку, можливо групування в категорії. Ці метадані живуть поряд з тілом статті, щоб і навігація сторінки, і посилання AI на розділ будувалися з одного джерела.

Інваріант: slug статті унікальний у межах help-center і стабільний (зміна slug ламає зовнішні посилання і SEO). Текст, який бачить користувач на сторінці, і текст, який отримує AI, походять з одного й того самого файлу.

## API і контракти

### Публічний контент довідки
Рендер статей серверний. Якщо контент читається через API (а не імпортується web-білдом напряму), це публічний unauthed read-only endpoint у тій самій зоні, що інші публічні сторінки. Спосіб доставки тексту в рендер, web-білд vs runtime-fetch, вирішується при імплементації; вимога плану, джерело тексту одне.

### Публічний AI-чат
Окремий публічний endpoint під anon-зоною, без JWT. Запит несе повідомлення користувача і клієнтську історію розмови. Відповідь, SSE-стрім токенів, симетрично існуючому формату подій кабінетного чату (token, done, error), без поля балансу. Захищений власним IP-rate-limit і глобальним денним бюджетом. На вичерпання бюджету або ліміту повертає машиночитабельний код, який фронт показує як degradation-стан з відсиланням на статику.

Жодне поле IBAN/taxId чи інших реквізитів не є частиною цього контракту, чат працює лише з базою знань довідки.

## UI і UX

### Help-center
- Індекс `/help`: список розділів з навігацією, короткими описами, м'яким CTA на реєстрацію.
- Стаття `/help/<slug>`: тіло статті, навігація між розділами, посилання на суміжні теми, той самий CTA.
- Публічний layout (anon-хедер/футер як на лендінгу), не кабінетний.
- Мобільно-first, темна/світла тема, доступність, за конвенціями проєкту.

### AI-чат
- Переюз чат-UI з кабінету (повідомлення, стрім, стан вводу), адаптований під anon і під вбудовування в сторінку довідки.
- Empty-state: підказка, що можна спитати, приклади запитань про продукт.
- Streaming-стан: токени з'являються поступово.
- Degradation-стан: коли бюджет/ліміт вичерпано, чат показує спокійне повідомлення і веде до читання статей. Не виглядає як помилка-збій.
- Off-topic/refuse: коли AI відмовляє у податковому питанні, це нормальна відповідь у потоці, не error-стан.
- Rate-limit для аноніма: зрозуміле повідомлення без технічних кодів.

### Anti-hallucination у відповідях
Коли в базі знань немає відповіді, AI прямо це каже і пропонує суміжний розділ або реєстрацію, не вигадує функціонал. Це поведінка промпту, але вона видима в UX і входить у DoD.

## Ризики

- **Витрати на anon-AI.** Публічний LLM-endpoint, це відкритий гаманець. Мітигація закладена (IP-ліміт + денний бюджет + Haiku + короткий max-tokens), але реальні пороги треба підбирати за спостереженням після запуску. Денний бюджет має бути консервативним на старті.
- **Галюцинації у фінансовому контексті.** Невірна інструкція про гроші підриває довіру. Заземлення на базу знань + явна anti-hallucination поведінка обов'язкові, не nice-to-have. Потрібна жива перевірка на типових питаннях.
- **Scope-leak.** Користувачі тиснутимуть на податкові поради. Guardrail треба перевірити адверсарно (спроби витягти пораду в обхід), а не лише на чесних питаннях.
- **Drift при недбалому редагуванні.** Якщо хтось обійде єдине джерело і захардкодить текст у рендері чи в промпті, інваріант ламається. Архітектура має робити обхід незручним, а не лише забороняти на словах.
- **SEO-регрес при зміні slug.** Зміна URL статті після індексації ламає посилання. Slug-и треба вважати стабільними з моменту публікації.

## Definition of Done

- `finly.com.ua/help` відкривається анонімом без логіну: індекс + окремі статті за власними URL з робочою навігацією.
- Статичні сторінки рендеряться серверно і повністю читабельні навіть коли AI-чат вимкнено/недоступний.
- Текст статей і база знань AI походять з одного джерела, друга копія тексту відсутня.
- Публічний AI-чат працює без auth, не списує executions, нічого не пише в БД.
- AI відповідає тільки з бази знань, відсилає на конкретні розділи, чесно визнає відсутність відповіді.
- AI відмовляє у податкових/юридичних/off-topic питаннях з відсиланням до фахівця, перевірено адверсарно.
- Працює per-IP ліміт для аноніма і глобальний денний бюджет, при вичерпанні, graceful degradation на статику.
- SEO-базис на місці (метатеги, canonical, sitemap для розділів).
- М'який CTA на реєстрацію присутній на сторінці і доречний у чаті.
- Доданий пункт у `docs/manual-checks` (anon-доступ, scope-guardrail, degradation, перевірка на живих питаннях).
