# Sprint 3 — Журнал рішень планування

> **Статус:** **постійний артефакт**, не тимчасовий чек-лист.
>
> Документ фіксує продуктові і технічні рішення, прийняті під час планування Sprint 3, з обґрунтуванням і закритими питаннями. Виконує роль, аналогічну `docs/product/qr-decisions.md` для QR-стандарту. На нього посилаються runtime-константи (`packages/types/src/constants/reserved-slugs.ts`) і майбутній `docs/sprints/03-cabinet-public/README.md`.
>
> Кожен пункт у форматі «питання → варіанти → рішення». Закриті — позначаються ✅ + дата + rationale. Журнал змін — внизу.

## A. Архітектурні блокери (закрити **до** старту коду)

- [x] **A1.** ✅ 2026-05-02 — **один проект**, host-aware routing. Cookie входу залишається на host-level (`finly.com.ua`, без leading dot), щоб публічна зона не бачила сесію. Google-вхід — на кабінетному домені.

- [x] **A2.** ✅ 2026-05-02 — **дві кнопки**: «Інший банк» (нова адреса `qr.bank.gov.ua`) + «Інший банк (запасний варіант)» (стара адреса `bank.gov.ua/qr`). Обидві адреси зашиті в коді як константи (env-змінна `NBU_PAYLOAD_LINK_HOST` зникає). QR-6 з блокера запуску перетворюється на post-launch перевірку: за реальним feedback зрозуміємо, чи варто прибрати запасну. Старий формат (002) як був hardcoded на `bank.gov.ua/qr` — без змін.

- [x] **A3.** ✅ 2026-05-02 — **тільки вивіска**: QR + кнопки банків. Клієнт сам вписує суму в банк-додатку. Список рахунків бізнесу не показуємо (privacy-ризик). Конкретний рахунок завжди має окреме посилання `pay.finly.com.ua/{business}/{invoice}` (Sprint 4) — ФОП сам надсилає його клієнту.

## B. UX-питання (закрити на kickoff, до верстки)

- [x] **B1.** ✅ 2026-05-02 — **11 логотипів банків неактивні** (сірі, не клікабельні) з підписом «Незабаром». Активні лише дві кнопки внизу: «Інший банк» (нова адреса) + «Інший банк (запасний варіант)» (стара адреса). Візуально показуємо майбутній стан, поведінково чесно.

- [x] **B2.** ✅ 2026-05-02 — **варіант 1 + 3 разом**: та сама верстка під toggle всередині кабінету (основний preview, ФОП не виходить з контексту) + кнопка «Відкрити в новій вкладці» (додатково для перевірки "як насправді"). Iframe з реальною публічною сторінкою всередині не робимо — зайва складність.

- [x] **B3.** ✅ 2026-05-02 — **URL-фундамент незмінний:** `pay.finly.com.ua/{business}` (вивіска) і `pay.finly.com.ua/{business}/{invoice}` (рахунок). Slug бізнесу:
  - **Free:** випадкові 8 символів (приклад: `aB3xQ9k7`), не редагується.
  - **Paid:** ФОП обирає (приклад: `ivanenko`).

  Для рахунку (Sprint 4): Free — теж випадкові 8 символів, Paid — формат на вибір ФОП. Залишається open question на Sprint 4: чи примусово додавати випадковий шматочок навіть у Paid (захист від послідовного підбору `zamovlennia-1`, `-2`, ...) або обмежитись попередженням ФОП-у. Зараз не вирішуємо.

- [x] **B4.** ✅ 2026-05-02 — **не закладаємо нічого під майбутнє**. Кабінет бізнесу у Sprint 3 — проста сторінка (налаштування + QR), без вкладок-заглушок. UI під рахунки узгоджуємо в Sprint 4 цілісно.

- [x] **B5.** ✅ 2026-05-03 — **реальні логотипи банків**. Якщо колись хтось із банків попросить прибрати — заміняємо на generic. Юридичний ризик мінімальний поки не позиціонуємо себе як партнера банку.

- [x] **B6.** ✅ 2026-05-03 — **за замовчуванням всі 11 банків увімкнені** + **мінімум 1 обов'язковий** (не можна зняти всі). Лінивий ФОП отримує максимальне покриття без зусиль; активний — налаштовує під себе; нульовий стан неможливий.

## C. Технічні питання (зафіксувати в плані спринту)

- [x] **C1.** ✅ 2026-05-03 — **два окремі поля**: `taxationSystem` (enum: `simplified-1`, `simplified-2`, `simplified-3`, `general`) + `isVatPayer` (boolean). Семантично точно (можна бути на спрощеній-3 + ПДВ). Валідаційне правило: `isVatPayer === true` дозволено лише при `simplified-3` або `general`. Юр-ревʼю Sprint 6 структуру не змінить — лише підтвердить.

- [x] **C2.** ✅ 2026-05-03 — **hard-delete одразу, без soft-delete**. Видалення бізнесу:
  - Confirm dialog перед видаленням.
  - Попередження, якщо у бізнесу є активні рахунки ("видаливши, клієнти з посиланнями не зможуть оплатити").
  - "Undo" toast з таймером 5 секунд (стандартний Gmail-style паттерн).
  - Hard-delete одразу — назавжди.
  - Slug звільняється одразу.

  Поле `deletedAt` у схемі (Sprint 1 заклав "на майбутнє") залишаємо невикористаним — коштує нуль, дає гнучкість якщо колись передумаємо. Soft-delete фіча (restore, корзина) — overengineering для MVP.

- [x] **C3.** ✅ 2026-05-03 — **розширюємо до повноцінного категоризованого списку**. Файл `packages/types/src/constants/reserved-slugs.ts` оновлений з 5 категоріями: TECHNICAL (12), SYSTEM (35), GOVERNMENT (17), BRANDS (111), PUBLIC_FIGURES (26). Загалом 201 зарезервоване ім'я. Розширюється точково по запиту/інциденту, без міграцій. **Окрема задача (не блокер):** нецензурщина як окремий файл з leetspeak-варіантами + open-source список. **Конверсійна логіка:** при колізії з reserved — повідомлення "зверніться до підтримки" → канал sales-конверсії для правовласників бренду.

- [x] **C4.** ✅ 2026-05-03 — **whitelist полів публічного endpoint**: `type`, `name`, `slug`, `acceptedBanks`. Все інше (IBAN, taxId, taxationSystem, isVatPayer, paymentPurposeTemplate, ownerId, managers, createdAt) **не віддається** в JSON. Реквізити доступні клієнту тільки через QR (відкривається банк-додатком). Найбезпечніший шлях: відсутність дублюючих каналів витоку.

- [x] **C5.** ✅ 2026-05-03 — **знак гривні в центрі обох QR** (нормативно правильно для payment-QR + візуальна узгодженість для URL-QR). **Лого Finly — як окремий UI-елемент під QR** (у верстці сторінки, не всередині картинки).

  **Хвіст Sprint 2 (треба виправити в Sprint 3):**
  - Замінити `apps/api/src/modules/qr/assets/finly-logo-qr.png` → `hryvnia-symbol.png` за зразком з PDF постанови НБУ № 97 (білий круг з знаком ₴, розмір — нормативний; норматив §II.11-12 ст. 5).
  - `QrLogoCompositor` стає `QrCenterSymbolCompositor` (або зберігається назва, але семантично — знак гривні, не довільне лого).
  - QR-4 manual check (`docs/manual-checks/README.md`) переформулюємо: "QR зі знаком гривні в центрі зчитується банк-додатками", без згадки "лого Finly".
  - Лого Finly додається у верстці кабінету і публічної сторінки як SVG/PNG під QR — це задача Sprint 3 frontend-частини.

  **Custom-logo бізнесу (Paid фіча, Sprint 6) лишається в плані** — теж піде в центр, замість знаку гривні. Це знаходиться у конфлікті з нормативом, але так було вирішено бізнесово ще в qr-decisions §2.2 (трактуємо як "ФОП-вибір з юридичним усвідомленням").

- [x] **C6.** ✅ 2026-05-03 — **знято**. Між Sprint 3 і Sprint 6 деплой на прод не планується — реальних ФОП з кількома бізнесами не існуватиме. Grandfather-логіки і feature-flag-у на ліміт у Sprint 3 **не закладаємо взагалі**. Sprint 6 додасть ліміт як звичайну валідацію — без додаткових міграцій / `if`-ів для legacy-юзерів.

- [x] **C7.** ✅ 2026-05-02 — закрито разом з A2. Адреси hardcoded як константи; env зникає.

- [x] **C8.** ✅ 2026-05-03 — **покроково (4 кроки)** з валідацією на кожному кроці і **step-навігатором зверху** з підсвічуванням поточного кроку (ФОП бачить весь шлях і "де я"):
  - **Крок 1:** Тип + Назва
  - **Крок 2:** IBAN + ІПН
  - **Крок 3:** Оподаткування + ПДВ
  - **Крок 4:** Призначення платежу + Банки

  Кліком по попередньому кроку у навігаторі — повертаємось редагувати. Перехід на наступний крок — лише після успішної валідації поточного.

## D. Свідомо поза скоупом Sprint 3

- Інвойси (Sprint 4).
- Індивідуальні per-bank кнопки (Sprint 5).
- Free vs Paid ліміти, кастомний логотип у QR (Sprint 6).
- Preview-mode для сторінки інвойсу (Sprint 4 разом з інвойсами).
- UI «вкладки Постійний QR / Рахунки» (Sprint 4 разом з інвойсами).
- Передача бізнесу між акаунтами, запрошення керівника (Phase 1.5+).
