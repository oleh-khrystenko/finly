# Upload Media — Sprint Plan

> **Ціль:** Додати можливість завантаження, редагування та видалення аватарки на сторінці профілю. Це перший media-функціонал платформи, який демонструє клієнтам готову інтеграцію зі storage-інфраструктурою (Cloudflare R2) та професійний підхід до роботи з медіа.

---

## Що будуємо і навіщо

CyanShip — демо-бойлерплейт для стартапів. Клієнт відкриває профіль, бачить свою аватарку (або заглушку з ініціалами), натискає — і отримує сучасний інтерфейс для вибору, обрізки та завантаження фото. Аватарка одразу з'являється в хедері, на дашборді — всюди де є компонент UiAvatar. Це живий proof що платформа вже має media pipeline: upload → CDN → instant delivery.

**Що вже готово в кодовій базі:**
- Поле `profile.avatar` (URL) в user schema — і в MongoDB, і в Zod контрактах
- Компонент `UiAvatar` — показує зображення або fallback на ініціали, обробляє помилки завантаження
- Хедер, дашборд, мобільне меню — вже відображають аватарку якщо URL є
- `UpdateProfileSchema` — вже приймає `avatar: z.string().url().optional()`
- Storage модуль — scaffold без реалізації, готовий до наповнення

**Що потрібно побудувати:** UI для вибору і обрізки, storage service (Cloudflare R2), endpoint для presigned URL, міграцію Google OAuth аватарки з зовнішнього URL на R2.

---

## User Flow: Встановлення аватарки (перший раз)

Користувач на сторінці профілю бачить свою аватарку — або заглушку з ініціалами якщо аватарки ще немає.

### Крок 1: Ініціація

Користувач натискає на область аватарки. З'являється overlay-іконка (камера або олівець) при наведенні — стандартний патерн, який одразу підказує що тут можна щось змінити. Клік відкриває діалог для роботи з аватаркою.

### Крок 2: Вибір зображення

В діалозі — зона для drop файлу або кнопка "Вибрати файл". Drag & drop є обов'язковим — це сучасний стандарт. Підтримуються формати: JPEG, PNG, WebP. Максимальний розмір вхідного файлу — 5 MB. HEIC свідомо не у списку: iOS Safari ≥14 автоматично конвертує HEIC → JPEG при виборі файла з Photos, якщо `accept` не містить HEIC MIME, тож iPhone UX зберігається без shipping'у WASM-декодера.

### Крок 3: Обрізка та масштабування

Після вибору файлу відкривається crop-інтерфейс прямо в тому ж діалозі:
- **Кругла маска** — стандарт для аватарок, одразу показує як виглядатиме результат
- **Zoom** — слайдер під зображенням для збільшення/зменшення
- **Drag** — перетягування зображення всередині рамки щоб вибрати потрібну область
- **Pinch-to-zoom** на мобільних — обов'язково, велика частина користувачів на телефонах

Це реалізується через бібліотеку **react-easy-crop** — стабільна, активно підтримується (500K+ завантажень/тиждень), підтримує `cropShape="round"`, мобільні жести, та має мінімальний API без зайвих залежностей.

### Крок 4: Підтвердження та завантаження

Кнопки "Зберегти" та "Скасувати". При натисканні "Зберегти":
1. Зображення обрізається на клієнті (canvas API) до квадрату фіксованого розміру
2. Конвертується в оптимальний формат (WebP для зменшення розміру)
3. Завантажується напряму в Cloudflare R2 через presigned URL (API сервер не проксює файл)
4. URL збереженого зображення записується в профіль користувача
5. Аватарка миттєво оновлюється скрізь в інтерфейсі

Під час завантаження — loading-стан на кнопці "Зберегти" і прогрес-індикатор. Помилки (файл занадто великий, мережева помилка) показуються як toast-повідомлення відповідно до tone конвенції.

## User Flow: Заміна аватарки

Коли аватарка вже встановлена, клік на неї відкриває діалог з двома опціями:
- **"Завантажити нову"** — запускає той самий flow вибору → обрізки → збереження. Стара аватарка видаляється з R2 після успішного збереження нової (не до — щоб не залишити користувача без аватарки при помилці).
- **"Видалити"** — окрема кнопка (див. наступний розділ).

Заміна — це не окремий flow, а той самий flow встановлення. Різниця лише в тому, що після успіху потрібно прибрати старий файл зі storage. Якщо стара аватарка — зовнішній URL (наприклад, Google), а не файл в R2 — крок видалення зі storage пропускається.

---

## User Flow: Видалення аватарки

Кнопка "Видалити аватарку" в діалозі (доступна тільки коли аватарка є). При натисканні:
1. Confirmation діалог: "Видалити аватарку?" з кнопками "Видалити" / "Скасувати" — стандартний `UiConfirmDialog`, вже є в системі
2. Після підтвердження: видалення файлу з R2 + очищення `profile.avatar` в базі
3. Інтерфейс одразу показує fallback — ініціали користувача (вже працює в `UiAvatar`)

## Google OAuth аватарка

### Поточна поведінка

Google strategy вже витягує аватарку з `profile.photos[0].value` і передає її в `UsersService.findOrCreateByGoogle()`, який зберігає URL в `profile.avatar` при створенні нового користувача. Тобто Google-аватарка вже працює — але як зовнішній URL (посилання на `lh3.googleusercontent.com`).

### Що змінюється

Зовнішній Google URL має ризики: може змінитись, стати недоступним, або Google може обмежити доступ. Тому при реєстрації через Google додається один крок:
1. Сервер отримує URL аватарки від Google (як і зараз)
2. **Нове:** зображення re-upload'ється на Cloudflare R2 — наша копія на CDN, повний контроль
3. URL зі storage (а не від Google) зберігається в `profile.avatar`

Після цього аватарка живе за загальними правилами — користувач може замінити або видалити її. Повернення до Google-аватарки після видалення не відбувається — fallback завжди на ініціали.

При повторному вході через Google (login, не signup) — аватарка не перезаписується. Якщо користувач вже встановив свою кастомну аватарку або видалив Google-аватарку — його вибір зберігається.

### Існуючі користувачі з Google URL

Користувачі, які зареєструвались через Google до цього спринту, мають зовнішній `lh3.googleusercontent.com` URL в `profile.avatar`. Окрема міграція не потрібна — вони продовжують працювати як раніше. Re-upload на R2 відбудеться природним шляхом: коли користувач вперше змінить аватарку через новий UI, нова буде на R2, а стара Google-посилання просто перезапишеться (видалення зі storage пропускається для зовнішніх URL).

## Storage: Cloudflare R2

### Чому R2

CyanShip вже використовує Cloudflare (Turnstile для CAPTCHA, Stream для відео). R2 — природне розширення:
- **$0 за egress** — найбільша перевага для media-heavy застосунків
- **S3-сумісний API** — стандартний AWS SDK, величезна екосистема
- **Presigned URLs** — direct upload/download без навантаження на API сервер
- **Єдиний billing** — не додаємо нового vendor'а

### Як працює upload

```
Клієнт запитує presigned URL → API генерує URL (R2 SDK) → 
Клієнт завантажує файл напряму в R2 → 
Клієнт повідомляє API про успіх → API зберігає URL в профіль
```

API сервер ніколи не проксює файли. Він тільки генерує presigned URL'и та зберігає metadata. Це стандартний production-патерн для будь-якого масштабу.

### Організація файлів в bucket

```
avatars/{userId}/{uuid}.webp
```

Один активний файл на користувача (старий видаляється при заміні). UUID у назві — для cache busting (кожен новий upload отримує новий URL, CDN не віддає стару версію). UUID обраний замість timestamp, бо timestamp вразливий до колізій при паралельних завантаженнях з кількох вкладок.

## Прийняті рішення

| Питання | Рішення |
|---------|---------|
| Максимальний розмір файлу | **5 MB** — покриває будь-яке фото з телефону, після crop + WebP фінальний файл ~50-200 KB |
| Роздільна здатність crop | **512×512 px** — чітко на Retina, компактний файл (на рівні Slack, GitHub, Discord) |
| HEIC (iPhone) | **Не підтримується на bundle-рівні.** Усі browser-side HEIC-декодери (`heic-to`, `heic2any`, `libheif-js`) транзитивно спираються на libheif (LGPL-3.0), що несумісне з permissive-профілем репо (ISC/MIT/UNLICENSED). iOS Safari ≥14 нативно конвертує HEIC → JPEG у file picker, коли `accept` не містить `image/heic`, — тож iPhone UX збережений без ship'у декодера. Користувачі з HEIC на Chrome/Firefox Desktop отримують toast `unsupported_format`. Trade-off свідомий: MVP уникає copyleft-obligations на користь простої legal-картини. Якщо у майбутньому з'явиться політика допустимості LGPL-3.0 — природний апгрейд: server-side декодування через `sharp` + libheif (LGPL на сервері допустимий для proprietary SaaS, не відображається у client bundle) |
| R2 bucket | **Один bucket `cyanship-media`**, папка `avatars/` — розділення можливе пізніше за потреби. Public access через Cloudflare CDN |
| Видалення Google-аватарки | **Fallback на ініціали** — без повернення до Google-фото, один простий flow для всіх випадків |
| Джерело правди розміру | **Application layer: client-side pre-check + HeadObject перевірка при commit з cleanup** — не transport layer. Причина: signed Content-Length у presigned PUT неможливий як upper bound (forbidden header у Fetch + exact-match семантика у S3 signature). Альтернатива (presigned POST з `content-length-range` policy) — overkill для малих аватарок під auth |
| Google re-upload timing | **Синхронно в OAuth callback** — додає 300-800ms до redirect, але уникає UX-стрибка URL після первинного рендеру |

## Known Limitations (MVP)

Свідомо прийняті trade-offs на поточному етапі. Документуються тут щоб їх не переосмислювали під час імплементації і щоб задати roadmap для наступних спринтів.

### 1. Orphaned файли в R2

**Сценарій:** Користувач завантажив файл через presigned URL, але commit не відбувся (мережева помилка, закрив вкладку). Файл залишається в `avatars/{userId}/` без прив'язки до профілю в БД.

**Чому acceptable для MVP:** Аватарка після crop+WebP — 50-200 KB. Навіть при 10% failure rate і 10 000 користувачах це <200 MB orphaned даних. R2 storage коштує $0.015/GB/місяць.

**Roadmap:** При масштабуванні або перед іншими media-типами (де файли великі) — додати TTL cron який сканує `avatars/` і видаляє файли без відповідного запису в БД (звірка через `user.profile.avatar`). Альтернатива — lifecycle policy на рівні R2 bucket з автоматичним видаленням файлів старших ніж N годин, якщо вони не tag'нуті як committed. Останній варіант чистіший бо не вимагає cron.

### 2. OAuth callback затримка через синхронний re-upload

**Сценарій:** Перший signup або login з Google callback → сервер робить `fetch(googleAvatarUrl) + sharp.resize + uploadBuffer` до redirect на `/auth/callback`. Додає 300-800ms до callback.

**Чому acceptable для MVP:** OAuth callback вже має значну затримку (JWT generation, session setup). 300-800ms — у межах психологічного порогу "повільного логіна". Альтернативи (fire-and-forget + асинхронне оновлення profile, job queue) створюють UX-стрибок URL: користувач бачить Google URL кілька секунд після логіна, потім URL замінюється на R2 при наступному refresh. Це гірший UX ніж одноразова затримка.

**Roadmap:** Якщо в майбутньому з'явиться job queue (для інших use cases) — перенести re-upload туди, і додавати на фронті subscription на profile updates (websocket або polling), щоб бачити оновлення avatar без refresh. Але це оверінжинірінг для поточної задачі.

### 3. Size enforcement не на transport layer

**Сценарій:** Ліміт на розмір файлу (5 MB) працює не через підпис presigned URL, а на application layer — через три шари захисту: клієнтська JS-валідація перед upload, серверна перевірка `HeadObject` при commit із cleanup'ом через `deleteObject` при rejection, rate limit на endpoint генерації presigned URL. Зловмисник з DevTools може пропустити клієнтський шар і upload'ити у свій namespace файл будь-якого розміру — commit його зловить і видалить, але файл на декілька секунд "живе" у bucket.

**Чому не signed `ContentLength`:** Технічно неможливо реалізувати upper-bound контроль через presigned PUT:
- `Content-Length` — forbidden request header у Fetch API (MDN). Браузер встановлює його автоматично з blob body, клієнт не може керувати через `fetch()`
- Signed `ContentLength` у PUT signature означає **exact match**, не upper bound (canonical request S3/R2 включає значення буквально). Upper-bound семантика через `content-length-range` — це фіча presigned POST (policy-based), не PUT

**Чому не presigned POST:** Справжній bucket-level upper bound реалізується через `@aws-sdk/s3-presigned-post` з policy, але це мультипартний `multipart/form-data` замість простого PUT з blob body. Overkill для малих приватних аватарок, правильний шлях для масштабування на великі публічні media-типи.

**Attack surface на поточному дизайні:** authenticated user може витратити кілька секунд R2 storage у своєму namespace. Ingress у Cloudflare R2 безкоштовний, cleanup відбувається синхронно в commit. Rate limit обмежує частоту. Maximum practical damage — негативна, не впливає на інших користувачів.

**Roadmap:** При додаванні media-типів за межами приватних аватарок (публічні зображення, відео) — міграція на presigned POST з `content-length-range` як первинний захист.
