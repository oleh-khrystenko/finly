# Sprint 14 — QR-брендинг: питання для уточнення

Статус: збір відповідей.

Контекст: два типи QR (НБУ-payload веде в банк-додаток; наш URL-QR веде на `pay.finly.com.ua`).
Колір обох вирішено лишити чорно-білим. Розрізнення несуть центр + рамки навколо коду.
Знак гривні (нормативний центр НБУ-QR) уже замінено на симетричний path з Wikimedia.

---

## Q1. Межа скоупу цього спринту

Уся ідея велика: рамки тип-1, центр+рамки тип-2 з двома форматами, монетизація (клієнтські
лого/назва/слоган + завантаження + модерація), QR на рівні бізнесу, download для друку.
Це 2-3 спринти. Що входить саме в цей?

Гіпотеза: MVP = базовий редизайн з Finly-брендингом (рамки + два дефолтні формати тип-2),
БЕЗ монетизації та клієнтського завантаження.

> Відповідь: Так, цей спринт = A (каркас рендеру рамок) + B (Finly-брендинг за замовчуванням).
> Шар C (монетизація) окремо. Важливо: платіжка з усіма платними функціями це окрема велика
> історія, що робиться окремо. Цей спринт лише закладає архітектуру (точки розширення) так,
> щоб потім вмикати платні фічі без переробок.

---

## Q2. Custom-branding (Paid-шар)

Клієнтські лого/назва/слоган із завантаженням через R2 + модерація. Цей спринт чи наступний?

> Відповідь: Поза скоупом. Частина громіздкої платіжно-монетизаційної історії (окремо).
> Цей спринт лише робить каркас extension-friendly: рендер не має зашивати Finly-asset як
> єдиний можливий, модель даних не має блокувати майбутні клієнтські поля.

---

## Q3. QR на рівні бізнесу + активація тип-2 в UI

Зараз URL-QR (тип-2) існує в API для рахунку/інвойсу, але ніде не показується. На рівні бізнесу
його немає взагалі. Цей спринт виводить тип-2 у видимий UI і додає рівень бізнесу?

> Відповідь: Так, усі три рівні.
>
> - Бізнес: тільки тип-2 (QR на сторінку-вітрину з рахунками); тип-1 неможливий (нема IBAN).
> - Рахунок: обидва (тип-1 плати + тип-2 відкрий сторінку).
> - Інвойс: обидва.
>   Тип-2 виводиться з orphan-стану у видимий UI на cabinet + public для всіх трьох рівнів.

---

## Q4. Download для друку

Уся теза монетизації спирається на "роздрукувати вивіску". Чи входить кнопка завантаження
QR як PNG-вивіски в цей спринт?

> Відповідь: Так, входить. Реалізація через параметр розміру на тому самому endpoint
> (QR не зберігається, рендериться на льоту з HTTP-кешем). Екран тягне дефолтний помірний
> розмір; кнопка завантаження веде на той самий endpoint з друкарським розміром. Не два
> окремі файли, а один генератор з різним розміром. Кеш працює окремо per-розмір.

---

## Q5. Два формати тип-2 (квадрат / прямокутник) у MVP

Без монетизації юзеру нічого завантажувати. Що рендеримо за замовчуванням: один формат
(квадрат-лого) чи юзер уже зараз обирає між двома? Чи це впливає на модель даних.

> Відповідь: MVP показує один фіксований дефолт тип-2 = Finly лого + назва (прямокутник).
> Перемикач квадрат/прямокутник і жорстка валідація завантаження це шар C (монетизація), не
> зараз. Але каркас рендеру (шар A) вміє обидва формати вже зараз, обидва під round-trip
> тестами, щоб шар C просто увімкнув вибір без переробки. Без нового поля в моделі даних у MVP.

## Q6. Тексти рамок у MVP

"Створено за стандартами НБУ" (верх тип-1) і слоган Finly (низ). У MVP це фіксовані
константи, без можливості редагувати?

> Відповідь: Так, фіксовані константи в коді (редагування/заміна = шар C).
>
> - Тип-2, низ (слоган): "Веди справи, а не папери."
> - Тип-1, верх: "Створено за стандартами НБУ" (робоче формулювання, шліфуємо при імплементації).
> - Тип-1, низ: "Finly" біля лого.
>   Увесь текст українською, без варіантів (single-locale uk).
