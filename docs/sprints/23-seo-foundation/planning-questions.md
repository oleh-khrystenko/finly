# Planning questions — SEO suite (Sprints 23+)

Покриває весь SEO-беклог з аудиту (15 задач). Розбивка спочатку запропонована як
A (тех-фундамент) / B (E-E-A-T + structured data) / C (перелінкування) / D (семантика+контент).
Нижче — лише рішення, що змінюють архітектурну форму планів. Дрібні технічні
деталі (формати іконок, точні поля schema) вирішуються при coding.

---

## Q1. Гранулярність: 3 спринти чи 4?

Блок C (перелінкування лендінг→help) — маленький і тематично зливається з
контентними роботами. Пропозиція: влити C у B (обидва чіпають help/лендінг-контент),
вийде 3 спринти: A (тех) / B (контент-довіра + перелінкування) / D (семантика+кластер).
Альтернатива — лишити 4 окремих, як запропоновано.

**Рекомендація:** 3 спринти.

**Відповідь:** 3 спринти. A (тех) / B (контент-довіра + перелінкування) / D (семантика+кластер).

---

## Q2. Авторство help-статей (E-E-A-T)

Зараз `Article`-schema без автора. Хто автор для пошуковика:
- варіант 1: Organization «Finly» (просто, але слабкий E-E-A-T);
- варіант 2: іменований експерт (бухгалтер-практик, що вже згаданий на лендінгу
  в блоці LandingPartner) як author/reviewer — сильніший сигнал довіри, але
  потребує мінімальної author-сутності (імʼя, роль, можливо коротке біо).

**Рекомендація:** варіант 2, якщо партнер згоден світитися як автор/рецензент;
інакше варіант 1.

**Відповідь:** Варіант 2. Автор — Тетяна Прядко, партнер-бухгалтер з лендінгу
(блок LandingPartner). Потрібна author-сутність (імʼя, роль, біо) + Person у schema.

---

## Q3. Search Console / статус індексації для семантичного спринта (D)

Спринт D (семантика + контент-кластер) спирається на реальні дані, не на здогади.
- finly.com.ua доданий у Google Search Console і вже накопичує запити/покази?
- маєш доступ, щоб поділитися експортом (реальні запити, CTR, позиції)?

Від цього залежить, чи D — «дослідження зараз» (є дані), чи «спершу зачекати на
накопичення + ручний аналіз видачі» (даних ще нема).

**Відповідь:** Сайт ще не в Search Console, історичних даних нема. Спринт D
починається з підключення GSC + ручного аналізу видачі; семантика будується на
SERP і намірі, пріоритети уточняться після накопичення даних.

---

## Рішення, які приймаю сам (не блокують форму)

- FAQ-секція + FAQPage-schema живе на лендінгу (там обробка заперечень конвертує),
  з дзеркалом на /help home за потреби.
- `SoftwareApplication.offers` показує реальні ціни каталогу (вони вже публічні
  на сторінці тарифів, ризику розкриття немає).
- privacy/terms лишаються `noindex` — це юридичний гейт (review юриста), поза
  скоупом SEO-спринтів; згадую в НЕ-скоупі.
- Растровий logo для structured data, host-aware robots, metadataBase, WebSite-node,
  auth titles, per-article OG — мехнічні, рішень не потребують.
