# Sprint 17 — міграція білінгу на WayForPay: уточнення

Робочий простір Фази 1. Питання, відповіді на які змінюють архітектурну форму плану. Фундамент фактів — `research-spike.md`.

## Підтверджені вхідні (провайдер-агностичні, НЕ перепитуються)

Зафіксовано власником до старту планування:

- Живих платних підписників у проді НЕМАЄ → чиста заміна (hard cutover), Stripe видаляємо повністю, без двопровайдерного співіснування, без міграції даних.
- У скоупі ОБИДВА типи оплат: підписка + разові пакети executions, мігруємо одразу.
- Каталог планів переїжджає зі Stripe Products у статичний типізований конфіг поруч з кодами планів. Без БД-колекції, без адмінки. Catalog-endpoint віддає той самий shape. Інваріант: єдине джерело істини для значення у 2+ місцях.
- Власний кабінет керування підпискою замінює Stripe Billing Portal. MVP: скасування (в кінці періоду), зміна плану, оновлення картки, мінімальний список останніх списань.
- Перехід плану: upgrade одразу з перерахунком executions, downgrade з наступного періоду, власний прапорець «запланована зміна з дати X». Без Stripe Subscription Schedules.
- Невдале регулярне списання: покладаємось на провайдерські ретраї (Модель A), вебхук → PAST_DUE + банер «онови картку», доступ до кінця періоду.
- Рекуренти: Модель A (нативні Regular payments) основна, Модель B (recToken+Charge) резерв. Остаточний вибір фіналізується у sandbox; абстракція не прибиває одну модель цвяхами.

## Відкриті питання (потребують рішення перед планом)

### Q1. Чи у скоупі повернення коштів (refund) цього спринту?

Поточний код refund не має (є лише dev-скидання білінгу). Spike підняв Refund як окремий флоу WayForPay. У MVP-переліку кабінету refund не згаданий.

Варіанти:

- Поза скоупом: повернення робимо вручну через кабінет WayForPay, у продукті не будуємо. Окремий спринт за потреби.
- У скоупі: будуємо refund-endpoint + кнопку (наприклад, для скасування з поверненням за невикористаний період).

Відповідь Q1: **У скоупі.** Будуємо refund-endpoint + UI. Скасування з поверненням за невикористаний період. Функціонал максимально user-friendly.

Відповідь:

### Q2. Як реалізуємо «оновлення картки» під Моделлю A?

WayForPay прив'язує картку при створенні регулярного платежу. Підтвердженого in-place card-update API spike не виявив (відкрите питання до sandbox). Тобто «змінити картку» технічно може означати скасувати поточний регулярний платіж і створити новий з повторним вводом картки (redirect на сторінку WayForPay).

Варіанти:

- Прийнятно: «оновити картку» = re-bind через новий checkout-redirect (під капотом cancel old + create new regular payment зі збереженням плану і дати). Користувач один раз вводить картку наново.
- Відкласти: прибрати «оновлення картки» з MVP кабінету до підтвердження можливостей у sandbox, лишити скасування + зміну плану.

Відповідь Q2: **Прийнятно: re-bind через checkout.** «Оновити картку» під капотом cancel old + create new regular payment зі збереженням плану і дати наступного списання. Користувач один раз вводить картку наново через redirect на WayForPay.

### Q3. Чи є у продукті безкоштовний пробний період (trial)?

Поточні enum-и містять TRIALING (scaffold зі Stripe). Це впливає на параметри створення підписки і на стан-машину.

Варіанти:

- Немає trial: підписка активна одразу після першої оплати. TRIALING-стан прибираємо як мертвий.
- Є trial: закладаємо N днів безкоштовно перед першим списанням.

Відповідь Q3: **Є trial, 1 місяць.** Підписка у стані TRIALING перший місяць, перше списання відкладене (через `dateBegin` у майбутньому). TRIALING-стан лишається живим. Sandbox-уточнення: як WayForPay в'яже/верифікує картку при відкладеному першому списанні (верифікаційне блокування коштів vs нульова прив'язка).

### Q4. Джерело «мінімального списку останніх списань» у кабінеті?

Наявний ledger (ExecutionTransaction) фіксує рух executions, не гроші. Список грошових списань потребує власного джерела.

Варіанти:

- Нова легка колекція платежів (сума, валюта, дата, статус), що наповнюється з вебхуків WayForPay. Кабінет читає останні N.
- Похідне з наявних записів про вебхук-події (ProcessedWebhookEvent), без нової колекції.
- Запит статусу в WayForPay на льоту (STATUS повертає lastPayedDate/nextPaymentDate, але не повну історію).

Відповідь Q4: **Нова легка колекція платежів.** Сума, валюта, дата, статус; наповнюється з вебхуків WayForPay. Кабінет читає останні N. Також природне джерело для refund (є запис, який повертаємо).

### Q5. Апгрейд плану: як стягуємо proration-доплату негайно?

Виявлено на plan-review. Обрано (вхідне): апгрейд застосовуємо одразу, proration-різницю за залишок періоду стягуємо ОКРЕМИМ негайним платежем, після чого `CHANGE` піднімає суму рекурента на наступний цикл. WayForPay `CHANGE` сам по собі негайної доплати не робить.

Підпитання, що міняє архітектуру: як саме проходить ця разова доплата під Моделлю A.

Контекст: у Моделі A (Regular payments) картку тримає WayForPay, у нас токена немає. Щоб стягнути довільну суму без участі клієнта (silent), потрібен `recToken` і списання через `Charge` host2host, тобто токенізація має бути увімкнена і токен збережений ще при створенні підписки (гібрид: розклад через Regular payments + токен для ad-hoc операцій). Без токена доплата = ще один redirect на сторінку WayForPay з повторним вводом картки посеред апгрейду.

Варіанти:

- Silent через збережений токен: при створенні підписки одразу беремо `recToken`, доплату за апгрейд стягуємо тихо через `Charge`. UX найкращий (апгрейд у один клік), але додає залежність: токенізацію треба активувати у WayForPay, і sandbox мусить підтвердити, що `recToken` доступний разом з Regular payments. Refund теж зручніше робити за збереженою транзакцією.
- Redirect Purchase: доплату проводимо окремим checkout-redirect, клієнт вводить картку наново. Простіше технічно, без токен-залежності, але апгрейд стає дворазовим вводом картки (гірший UX).

Поведінка при невдалій доплаті (обидва варіанти): апгрейд застосовуємо ТІЛЬКИ після підтвердженої оплати різниці. Невдача — план не змінюється, executions не нараховуються.

Відповідь Q5: **Silent через токен.** При створенні підписки беремо `recToken`, доплату за апгрейд стягуємо тихо через `Charge`. Це гібрид: Regular payments веде розклад, `recToken` для ad-hoc списань. Активація токенізації стає блокером дня один (поряд з активацією рекурентів). Sandbox мусить підтвердити, що `recToken` віддається разом зі створенням Regular payment. Апгрейд застосовується тільки після підтвердженої доплати.
