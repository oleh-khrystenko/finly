# Інструкція для Gemini Agent: Аудит та оновлення контексту

Виконай повний аудит кодової бази проєкту та онови файл контексту `GEMINI.md`.

## Філософія

`GEMINI.md` — це **навігаційна карта + правила**, НЕ документація і НЕ копія кодової бази.

Тест для кожного блоку: **"Чи агент прийме ГІРШЕ рішення без цієї інформації?"**
Якщо ні — не включай.

Агент може за секунди відкрити будь-який файл. Тому:
- НЕ копіюй зміст файлів (таблиці полів схем, enum values, списки UI компонентів)
- НЕ вставляй code blocks — вказуй шлях до файлу-прикладу
- НЕ дублюй те, що є в `docs/` — посилайся
- НЕ описуй те, що очевидно з назв файлів та структури директорій

## Ліміти

- **Цільовий розмір: до 30,000 символів. Жорсткий ліміт: 40,000.**
- Після генерації перевір розмір. Якщо > 40,000 — скороти, починаючи з найменш критичних секцій.

## Інструкції

### Крок 1 — Аналіз

Рекурсивно проскануй ВСЮ кодову базу. Зрозумій:
- **Архітектура:** Загальний підхід та як модулі пов'язані між собою.
- **Domain Model:** Ключові бізнес-сутності та схема БД.
- **Dependency Graph:** Хто від кого залежить (imports analysis).
- **Patterns:** Повторювані рішення (Authentication, Error Handling, Validation).
- **Config & Environment:** Які змінні критичні (env, config files).
- **Testing:** Стратегія тестування (E2E, Unit) та де вони знаходяться.

### Крок 2 — Збереження Manual Knowledge

Прочитай поточний `GEMINI.md`. Весь контент між маркерами `<!-- MANUAL:START -->` та `<!-- MANUAL:END -->` збережи в оновленому файлі **БЕЗ ЖОДНИХ ЗМІН** — дослівно. Це зона відповідальності людини.

### Крок 3 — Запис

Перепиши/допиши `GEMINI.md` з такою структурою:

---

# {Назва проєкту}
> Одне речення — технічна суть проєкту.

## Tech Stack
Тільки **ключові** технології (Framework, Language, DB, Auth, Payments, State, Styling, i18n, Tests).
НЕ включай дрібні утилітні бібліотеки.

## Architecture Overview
3–5 речень: архітектурний підхід, шари, статус модулів. Тільки структурні рішення, без переліку фіч.

## Project Structure
Дерево директорій, **максимум 2 рівні вкладеності** від кореня кожного app/package.
Анотації — максимум 3-5 слів після `#`.

## Domain Model & Schema
Для кожної моделі: назва + шлях до schema файлу + 2–3 ключові особливості (наприклад, soft-delete).
НЕ пиши перелік полів — агент прочитає файл.

## Module Dependency Map
Граф залежностей у компактному форматі:
`Module A` → `Module B`
`Module C` ↔ `Module D` (circular)

## Key Patterns (CodeDNA)
Для кожного патерну: **правило + шлях до файлу-прикладу**.
БЕЗ code blocks. Приклад: `Створення endpoint: Guard + DTO + Service. Файл: users.controller.ts`.

## API Surface
Згрупований список ендпоінтів: метод, шлях, guard, 3–5 слів опису.

## Environment & Config
Критичні env змінні, їх призначення та fail-fast policy. Помилки тут коштують найдорожче.

## Dev Workflow
Команди для: dev, build, lint, test, docker. Compact список.

## Rules & Conventions
⚠️ ЗБЕРЕЖИ З ПОТОЧНОГО `GEMINI.md` (між маркерами MANUAL).

## Known Complexities & Debt
**Тільки gotchas** — речі, які агент НЕ зрозуміє тільки з коду:
- Workarounds з поясненням ЧОМУ.
- Неочевидні залежності (наприклад, Stripe rawBody).
- Circular dependencies та їх рішення.
- Runtime поведінка, що відрізняється від очевидної.

### Вимоги до стилю
- **Лаконічність:** Висока щільність інформації, "сухий" техностриль.
- **Навігація:** Завжди вказуй точні шляхи до файлів.
- **Мова:** Українська (описи), Англійська (терміни, код).

### Антипатерни (НЕ роби цього)
- ❌ Таблиці з переліком полів моделей (агент відкриє schema).
- ❌ Таблиці UI компонентів (агент відкриє shared/ui).
- ❌ Code blocks з прикладами (достатньо шляху до файлу).
- ❌ Повний опис flow кроками (посилайся на `docs/architecture/`).
- ❌ Дублювання інформації між секціями.
