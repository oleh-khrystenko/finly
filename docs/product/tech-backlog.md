## Архітектурні «заготовки» в базі (закласти на старті, без UI)

Ці структури коштують копійки зараз і дорого — потім (міграції, ризик у проді). UI / повноцінну логіку під них **не робимо в MVP**, лише структуру.

- **Ролі користувачів** — три значення в БД: `Гість`, `Зареєстрований користувач`, `Адмін`. Логіку під «Адмін» зараз не пишемо взагалі — лише поле в БД.
- **«Режим бухгалтера» як прапорець на акаунті** — окреме булеве поле в профілі зареєстрованого користувача. Увімкнено = людина веде чужі бізнеси. Це **не окрема роль**: одна людина = один акаунт = одна верифікація (важливо для майбутнього KYC).
- **Сутність "Бізнес" із полем "тип"** — основна сутність із унікальною публічною сторінкою. Має поле «тип бізнесу» (юридична форма). Sprint 7 розширив enum з одного значення (ФОП) на 4: фізособа / ФОП / ТОВ / неприбуткова організація. Підтипи (ПрАТ-vs-ТОВ, ОСББ-vs-фонд) — підмножини одного з 4, не нові enum-значення.
- **Бізнес може існувати без власника** — це валідний стан (наприклад, людина в режимі бухгалтера створила бізнес для клієнта, який ще не зареєстрований). Поле «власник» одразу робиться опціональним.
- **Структура `managers` (керівники бізнесу)** — у MVP туди записується автор створення для бізнесів без власника. Це єдиний механізм доступу: «має доступ той, хто власник або є в керівниках». Запрошення інших людей у керівники — Phase 1.5+, без зламу схеми.

## Валідатори

- **ЄДРПОУ-checksum (ДКСУ-алгоритм)** — _пріоритет: low._ Sprint 7 §SP-2 свідомо валідовує ЄДРПОУ лише як `^\d{8}$` без checksum-перевірки. Реалізація відкладена з трьох причин: (1) ДКСУ-алгоритм має 2-фазну логіку (друге проходження з вагами 3..9 у разі залишку 10 на першому проході) і edge-cases (legacy-коди до 1992 для державних підприємств, нерезидентські коди, коди філій); naive-implementation відсіче ~5-10% валідних реальних ЄДРПОУ як false-negative. (2) ЄДРПОУ — публічний реєстр (`opendatabot` за 5 секунд); РНОКПП-checksum мав сенс через ручне введення, ЄДРПОУ зазвичай copy-paste з документа. (3) Реальний контроль робить банк-додаток клієнта при списанні (Finly = "тупий генератор", Модель А — `qr-decisions.md` §1.12). **Коли робити:** після збору реального feedback "ввели typo у ЄДРПОУ, банк відмовив" від ≥3 ФОП-клієнтів, або при появі explicit-вимоги від великого замовника. **Implementation note:** додавання checksum пізніше — non-breaking change (нові документи можуть мати помилку тільки коли writer ігнорує warning, що uncommon). Не йдемо у компроміс "checksum як warning без блоку submit" — це не патерн поточного codebase: всі Zod-validators блокуючі, RHF не має concept "warning issue", додання нового рівня помилок ламає universal `mapValidationCode` mapping. Файл-точка: `packages/types/src/validation/tax-id.ts` (`legalEntityTaxIdZod`).
