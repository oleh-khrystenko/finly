## Архітектурні «заготовки» в базі (закласти на старті, без UI)

Ці структури коштують копійки зараз і дорого — потім (міграції, ризик у проді). UI / повноцінну логіку під них **не робимо в MVP**, лише структуру.

- **Ролі користувачів** — три значення в БД: `Гість`, `Зареєстрований користувач`, `Адмін`. Логіку під «Адмін» зараз не пишемо взагалі — лише поле в БД.
- **«Режим бухгалтера» як прапорець на акаунті** — окреме булеве поле в профілі зареєстрованого користувача. Увімкнено = людина веде чужі бізнеси. Це **не окрема роль**: одна людина = один акаунт = одна верифікація (важливо для майбутнього KYC).
- **Сутність "Бізнес" із полем "тип"** — основна сутність із унікальною публічною сторінкою. Має поле «тип бізнесу» (юридична форма): у MVP єдине значення — ФОП; у майбутньому додаються ТОВ, ВАТ та інші. Закладаємо узагальнену схему одразу, навіть якщо UI лише під ФОП.
- **Бізнес може існувати без власника** — це валідний стан (наприклад, людина в режимі бухгалтера створила бізнес для клієнта, який ще не зареєстрований). Поле «власник» одразу робиться опціональним.
- **Структура `managers` (керівники бізнесу)** — у MVP туди записується автор створення для бізнесів без власника. Це єдиний механізм доступу: «має доступ той, хто власник або є в керівниках». Запрошення інших людей у керівники — Phase 1.5+, без зламу схеми.
