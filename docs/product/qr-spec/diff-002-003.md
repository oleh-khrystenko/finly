# Diff формату 002 → 003

> **Контекст:** обидва формати співіснують у Finly. 003 — основний (sprint plan §1.1), 002 — fallback для банків, що не оновили парсер. Цей файл — джерело rationale для коментарів у `packages/types/src/qr/payload-002.ts` vs `payload-003.ts`. Якщо у TypeScript-коді треба пояснити «чому в 002 поле X порожнє, а в 003 заповнене» — посилаємось сюди.
>
> **Джерело:** [`nbu-003-spec.pdf`](./nbu-003-spec.pdf), додатки 3 (002) і 4 (003). Деталі полів — [`README.md`](./README.md).

## Деталі по полях

### Гіперпосилання (header)

**002 (Додаток 3, таблиця 1, ст. 15):**

- `Код старту застосунку` — **23 B fixed**, два дозволені значення: `https://qr.bank.gov.ua/` або `https://bank.gov.ua/qr/`.
- `Закодована структура даних QR` — `≤475 B`, Base64URL.

**003 (Додаток 4, таблиця 1, ст. 23):**

- `Код старту застосунку` — **50 B max var**, дозволені значення: `https://qr.bank.gov.ua/` **або «персоніфікований код старту застосунку, який містить назву домену надавача платіжних послуг»** (§I.2.1, ст. 24).
- `Закодована структура даних QR` — `≤475 B`, Base64URL.

**Чому це принципово.** У 002 хост — closed enum з двох. У 003 НБУ відкрила можливість для НПП мати свій host (наприклад, `https://send.monobank.ua/qr/`), що дозволяє App Links / Universal Links відкривати конкретний банк без системного пікера. Це ж відкриває research-питання Sprint 5 §3.1 (per-bank deep-links). **Для Finly як не-НПП цей механізм не доступний** — наш host не може бути «персоніфікованим» у нормативному сенсі. Деталі — у [`README.md` секція "Host у нормативі"](./README.md#host-у-нормативі).

### Поле 4 — «Функція»

**002 (§II.4.4, ст. 19):** виключно `UCT` (Ukrainian Credit Transfer).

**003 (§II.4.4, ст. 27):** один з трьох:

- `UCT` — кредитовий переказ (стандартний),
- `ICT` — миттєвий кредитовий переказ (англ. Instant Credit Transfer),
- `XCT` — миттєвий АБО кредитовий (вибір клієнтом у банк-додатку).

**Builder-наслідок.** `payload-003.ts` приймає параметр `function: 'UCT' | 'ICT' | 'XCT'` з дефолтом `UCT`. У MVP Finly (Модель А — генератор без трекінгу оплат) `ICT` і `XCT` не потрібні; залишаємо як параметр для майбутньої еволюції в Phase 1.5.

### Поле 5 — рядок 5

**002:** «BIC» — Bank Identifier Code банку отримувача. Маркований як **R** (резервний, RFU). У payload завжди порожній.

**003:** «Унікальний ідентифікатор отримувача» — теж **R** (RFU). У payload завжди порожній.

**Висновок.** Косметичне перейменування поля. Обидва builder-и пишуть порожній рядок (`''`).

### Поле 10 — «Ціль» / «Категорія / ціль»

**002 (§II.4.10, ст. 19):** «Ціль» — 4 B, **R** (RFU). Завжди порожнє.

**003 (§II.4.10, ст. 27):** «Категорія / ціль» — 9 C, **O** (обовʼязкове). Формат `CCCC/PPPP` за класифікатором ISO 20022 (постанова НБУ № 93 від 16.09.2021).

Приклади з PDF:

- `SUPP/SUPP` — комунальні послуги (Додаток 4 §V.12 ст. 30).
- `MP2P/MP2B` — P2P-переказ (Додаток 4 §V.15 ст. 31).
- `MP2B/GSCB` — товари в магазині (Додаток 4 §V.18 ст. 33).
- `OTHR/GDDS` — товари в інтернет-магазині (Додаток 4 §V.21 ст. 34).

**Builder-наслідок.** `payload-003.ts` приймає параметр `categoryPurpose: string` (формат `CCCC/PPPP`). Дефолт для MVP Finly — підлягає вибору при імплементації §2.1; кандидат `OTHR/GDDS` для типового retail/services кейсу ФОП. Окремо — Zod-валідація формату через regex `/^[A-Z]{4}\/[A-Z]{4}$/` у `input.ts`.

### Поле 11 — «Reference»

**002 (§II.4.11, ст. 19):** 35 B, **R** (RFU). Завжди порожнє.

**003 (§II.4.11, ст. 27):** 35 B, **Оп.** Унікальний у системі отримувача ідентифікатор — звʼязує QR-зчитування з конкретним рахунком на оплату ФОП у його обліковій системі. У MVP Finly (Модель А, без трекінгу оплат) це поле **не потрібне** — ми не маємо backend-обліку рахунків. Залишаємо порожнім, але builder приймає опціональний параметр для Phase 1.5 (Модель Б, трекінг через відповідність `Reference` ↔ `invoice.id`).

### Поле 12 — «Призначення платежу»

**002 (§II.4.12, ст. 19):** 420 C, **O** (обовʼязкове). Звичайний текстовий опис.

**003 (§II.4.12, ст. 27):** 420 C, **O**. **Розширений синтаксис:** якщо текст починається з `?`, далі — key-value параметри у форматі `<name>="value"` через `&`. Приклад з PDF (§V.21, ст. 34):

```
?MerchantBusinessName="ROZETKA.UA", Покупка товарів, замовлення №821558965.
```

**Builder-наслідок.** Sprint 2 MVP **не використовує** `?`-синтаксис у 003 (нема джерела для key-value метаданих у Finly Модель А). Звичайний текстовий рендер працює в обох форматах однаково. Кодування `?`-синтаксису — еволюційна точка для Phase 1.5 (трекінг merchant ID, типу клієнта тощо).

### Поле 13 — «Відображення (дисплей)»

**002 (§II.4.13, ст. 19):** 70 C, **R** (RFU). Завжди порожнє.

**003 (§II.4.13, ст. 27):** 140 C, **Оп.** Текст для виведення на дисплей (не входить у дані переказу). Якщо починається з `?` — XML-фрагмент ISO 20022 з додатковими тегами платіжної інструкції (приклад `<UltmtCdtr>` з PDF §V.15 ст. 31).

**Builder-наслідок.** У MVP Finly не використовується. Залишаємо порожнім у 003.

### Поля 14–17 — нові у 003

Усі чотири — нові поля без аналога в 002.

**14. Код заборони зміни полів** (4 B, Оп., A) — hex bitmask 0..FFFF. Біт N = `1` забороняє редагування поля N платником. Біти **1–5, 11, 14–17 завжди = `1`** (норматив, §II.4.14, ст. 27). Приклади з PDF:

- `FEFF` — заборонено все, крім поля 8 (сума) (§V.12, ст. 29).
- `FFFF` — заборонено все (§V.18, ст. 33).

Для MVP Finly:

- `Invoice.amountLocked === false` → `FEFF` (клієнт може редагувати суму).
- `Invoice.amountLocked === true` → `FFFF` (нічого не редагується).
- `Business`-сторінка (без інвойсу, з нульовою сумою) → `FEFF` (клієнт обовʼязково вводить суму; решта зафіксована).

**15. Дата/час дії рахунку на оплату** (14 B, Оп., A) — формат `РРММДДГГХХСС`. Якщо порожнє — рахунок безстроковий. Маппиться на `Invoice.validUntil`:

- `validUntil === null` → порожнє.
- `validUntil instanceof Date` → форматуємо у `YYMMDDHHmmss` (UTC або Київський час — норматив не уточнює; ставимо UTC, бо `РР` = 2 цифри року може бути неоднозначним після 2099, але для MVP horizon-у це нерелевантно).

**16. Дата/час формування рахунку** (14 B, Оп., A) — теж `РРММДДГГХХСС`. Обовʼязкове, **якщо** заповнене поле 17 «Електронний підпис»; інакше опціональне. У MVP Finly підписи не використовуємо → поле 16 теж не заповнюємо.

**17. Електронний підпис даних** (90 B, Оп., A) — RFU. У MVP не використовуємо.

## Розділювач — нормативна різниця

**002 (§III.7–10, ст. 20):**

- Дозволені варіанти: `Lf` (0Ah) або `Cr+Lf` (0D 0Ah).
- Варіант фіксується одразу після `BCD` і **однаковий для всіх рядків**.
- Для мінімізації розміру **рекомендується `Lf`**.

**003 (§III.7, ст. 28):**

- **Виключно `Lf`** (норма, не рекомендація).

**Builder-наслідок.** Обидва builder-и можуть однаково використовувати `'\n'` (Lf) — для 002 це рекомендований варіант, для 003 єдиний дозволений. Спрощує код: один `payload.join('\n')` без розгалуження по версіях.

## Рівень корекції помилок — нова обмеженість для 003

**002 (§IV.10, ст. 21):** не уточнено явно — діє загальне правило з Додатка 1 §III.13 (ст. 5): з логотипом — `M` або `Q` (не `L`).

**003 (§IV.10.4, ст. 28):** **«рівень корекції помилок "M" або "Q"»** — явна вимога саме в процедурі генерації 003. `L` заборонено навіть за відсутності логотипу.

**Builder-наслідок.** `QrImageRenderer` для 003 має примусово `M` або `Q`. Sprint plan §2.3 каже «дефолт `H`» — це **відхиляється від нормативу**, бо `H` не у переліку (`M` або `Q`).

⚠️ **Конфлікт зі sprint plan §2.3.** Sprint plan вимагає `H` для logo overlay (~30% надлишковості). Норматив 003 — `M` або `Q` (15%/25%). Це **технічний компроміс** на стороні нормативу: лого = обовʼязкове, але рівень корекції обмежений `M`/`Q`. Розвʼязка:

- Якщо лого розміру ≤ 15% від QR — `M` норматив-OK і lo-go-readable.
- Якщо лого > 15% — порушуємо норматив. Sprint plan §2.3 вже фіксує `logoMaxRatio ≤ 0.30` — це **не нормативне число**.
- **Рішення для імплементації §2.3:** використовуємо `Q` як дефолт для 003 (макс рівень із дозволених, ~25% надлишковості) і обмежуємо `logoMaxRatio` до **0.20** (під 25%-стелю з запасом). Sprint plan §2.3 потребує оновлення з `H`/0.30 → `Q`/0.20 — задача §2.3, не §2.0. Sprint 2 §2.0 просто фіксує цей конфлікт тут.

## Що змінилось у нормативі загалом (vs попередній постанови № 68 від 28.05.2020)

Постанова № 97 від 19.08.2025 **скасовує** № 68 (28.05.2020) і № 11 (01.02.2021). Чинна з 01.11.2025.

З огляду на Sprint 2 ця skasovana історія не цікава для коду — `payload-002.ts` пишемо за чинним нормативом № 97 додаток 3, не за legacy. Якщо реальний банк-додаток виявиться застарілим парсером (не оновленим після 01.11.2025), QR-1/QR-2 manual checks це покажуть, і ми переключимо `BANK_PAYLOAD_VERSION` для відповідного банку у fallback `'002'`.
