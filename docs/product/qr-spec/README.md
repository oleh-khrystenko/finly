# QR-payload — нормативна специфікація НБУ

> **Статус:** активний артефакт, sprint 2 §2.0.
> **Джерело правди:** [`nbu-003-spec.pdf`](./nbu-003-spec.pdf) — постанова Правління НБУ від 19.08.2025 № 97 «Про затвердження Правил формування, передачі та обробки структури даних і графічного зображення QR-коду для обміну реквізитами кредитових та миттєвих кредитових переказів». Чинна з 01.11.2025. Скасовує постанови № 68 від 28.05.2020 і № 11 від 01.02.2021.
> **Як читати:** усі довжини, обовʼязковості, кодування, порядок полів — посилаються на конкретні сторінки PDF. Коли TypeScript-константа у `packages/types/src/qr/` мутить це значення, у коментарі обовʼязково посилання сюди (наприклад, `// see qr-spec/README.md §FIELD_PURPOSE`).

| 2   | Версія формату         | 3 B     | Fix     | O       | A         | `002`                                      |
| 3   | Кодування              | 1 B     | Fix     | O       | A         | `1` (UTF-8) або `2` (Win1251)              |
| 4   | Функція                | 3 B     | Fix     | O       | A         | `UCT` (Ukrainian Credit Transfer)          |
| 5   | BIC                    | 11 B    | Var     | **R**   | A         | RFU — порожнє (тільки `Lf`)                |
| 6   | Отримувач              | 140 C   | Var     | O       | \*        | ПІБ ФО / найменування ЮО                   |
| 7   | Рахунок отримувача     | 29 B    | Fix     | O       | A         | IBAN                                       |

| 12  | Призначення платежу    | 420 C   | Var     | O       | \*        | текст                                      |
| 13  | Відображення (дисплей) | 70 C    | Var     | **R**   | \*        | RFU — порожнє                              |

**Обмеження:**

- Загальний розмір даних до Base64URL: ≤ **507 байт** (§IV.11, ст. 21).
- Закодована структура (Base64URL): ≤ **475 байт** (таблиця 1).
- Максимальна QR-версія: **17** (85 модулів) (§IV.12, ст. 21).

### Сума / валюта — формат значення (§II.4.8, ст. 19)

- Префікс — 3 великі літери коду валюти. Єдина валюта для 002: `UAH`.
- Максимум: `999999999.99`.
- Десятковий розділювач: `.`. Якщо дрібної частини нема — `.` і нулі **опускаються** (`UAH3`, не `UAH3.00`).
- Якщо дрібна частина є — рівно 2 цифри (`UAH3.50`, не `UAH3.5`).
- Нулі перед цілою частиною заборонені.
- **Порожнє поле** = клієнт вводить суму при ініціюванні платежу.

### Кодові позначення (§III.6, ст. 20)

- **Обовʼязковість:** `O` — обовʼязковий (не може бути порожнім); `Оп.` — опціональний (присутній, може бути порожнім); `R` — резервний (мусить бути порожнім — тільки `Lf`).
- **Кодування:** `A` = ISO 646; `*` = ISO 646 або UTF-8/Win1251 згідно з полем «Кодування» (рядок 3). Усі `*`-поля мають однакове кодування.

### Розділювач рядків (§III.7–10, ст. 20)

- Дозволені: `Lf` або `Cr+Lf`.
- Варіант фіксується одразу після `BCD` і **однаковий для всіх рядків**.
- Для мінімізації розміру 002 рекомендується `Lf` (§III.10, ст. 20).
- Порожні поля = тільки `Lf` (без даних).

**Поле «Функція» (§II.4.4, ст. 27):**

- `UCT` — кредитовий переказ;
- `ICT` — миттєвий кредитовий переказ;
- `XCT` — миттєвий або кредитовий (вибір клієнтом).

Для MVP Finly (Модель А — генератор без трекінгу) дефолт `UCT`. Інші значення — еволюційна точка для майбутньої інтеграції з банками (Phase 1.5+, не зараз).

**Поле «Категорія/ціль» (§II.4.10, ст. 27):** обовʼязкове в 003, формат `CCCC/PPPP` за класифікатором ISO 20022 (постанова Правління НБУ № 93 від 16.09.2021). Приклади з PDF: `SUPP/SUPP` (комунальні послуги, ст. 30), `MP2P/MP2B` (P2P, ст. 31), `OTHR/GDDS` (товари, ст. 33). **Sprint 2 фіксує дефолт для MVP кейсу «оплата товарів/послуг ФОП»** — підлягає вибору при імплементації §2.1 (`packages/types/src/qr/payload-003.ts`); консервативний кандидат `OTHR/GDDS` (інше / товари), бо ФОП-кейси Finly у MVP — це переважно retail/services, а не комунальні платежі чи P2P.

**Поле «Код заборони зміни полів» (§II.4.14, ст. 27–28):** bitmask hex. Біт N = `1` забороняє платнику редагувати поле N. Біти **1–5, 11, 14–17 завжди мусять бути `1`** (примусова заборона). Приклади:

- `FEFF` — дозволено редагувати лише поле 8 (сума) (§V.12, ст. 29).
- `FFFF` — заборонено редагувати все (§V.18, ст. 33).

Для MVP Finly:

- Інвойс з `amountLocked: false` → `FEFF` (клієнт може правити суму).
- Інвойс з `amountLocked: true` → `FFFF` (клієнт нічого не редагує).

### Обмеження (§IV.8–9, ст. 28)

- Загальний розмір даних до Base64URL: ≤ **507 байт**.
- Закодована структура (Base64URL): ≤ **475 байт**.
- Максимальна QR-версія: **17** (85 модулів).

### Сума / валюта — формат значення (§II.4.8, ст. 27)

Ідентичний 002: префікс `UAH`, без zero-padding, мінімізація `UAH3` замість `UAH3.00`.

### Розділювач рядків (§III.7, ст. 28)

**Виключно `Lf`** для 003. Це **жорсткіша вимога**, ніж у 002 (де дозволені обидва варіанти). У builder-і 003 використовуємо лише `\n` (Lf, 0Ah).

## Деривація лімітів для Sprint 1 (`effectiveLimit`)

Sprint 2 §2.2 каже: Zod-обмеження `Business.name` і `Business.paymentPurposeTemplate` / `Invoice.paymentPurpose` беруть `MIN(FIELD_LIMITS[v].field)` по `PAYLOAD_VERSIONS`. Конкретні значення з нормативу:

| Поле в Finly                                                           | 002 (chars)       | 003 (chars) | MIN                             | Bytes (limit таблиці)                                                    |
| ---------------------------------------------------------------------- | ----------------- | ----------- | ------------------------------- | ------------------------------------------------------------------------ |
| `Business.name` (поле 6 «Отримувач»)                                   | 140               | 140         | **140**                         | n/a (ліміт у chars; bytes-обмеження виходить з загального розміру 507 B) |
| `Business.paymentPurposeTemplate` / `Invoice.paymentPurpose` (поле 12) | 420               | 420         | **420**                         | n/a                                                                      |
| Reference (поле 11)                                                    | n/a (R/RFU у 002) | 35 B        | n/a — у 002 не використовується | 35 B                                                                     |

**Висновок:** для `name` і `paymentPurpose` MIN дорівнює одному значенню (різниці між 002 і 003 нема). Sprint 1 hardcode `name.max(140)` і `paymentPurpose.max(420)` випадково збігся з нормативом — після §2.2 це деривується з `FIELD_LIMITS`, тобто стає захищеним від мовчазного дрейфу.

**Bytes-обмеження окремо.** Норматив оперує `chars` для текстових полів і `bytes` для сум/IBAN/кодів. Але є глобальне обмеження **507 байт на payload** (§IV.11 ст. 21 для 002; §IV.8 ст. 28 для 003). Це означає: якщо `name` = 140 chars кирилиці UTF-8 (~280 B) + `purpose` = 420 chars кирилиці UTF-8 (~840 B), сумарний payload **переповнить 507 B** і build впаде на per-version length-assertion. Це **очікувана поведінка**, не баг — UI Sprint 3 має показувати клієнту counter "залишок символів" з урахуванням реального байт-споживання.

## Host у нормативі

**Це блокувальне рішення §2.0 sprint plan.**

### Що норматив **fixує** (mandates)

**Формат 002 (Додаток 3, таблиця 1, ст. 15):**

> «Код старту застосунку» — **23 B, фіксована довжина**. Значення: `https://qr.bank.gov.ua/` **АБО** `https://bank.gov.ua/qr/`. **Інші хости специфікація не передбачає.**

**Формат 003 (Додаток 4, таблиця 1, ст. 23 + §I.2.1, ст. 24):**

> «Код старту застосунку» — **50 B max, змінна**. Значення: `https://qr.bank.gov.ua/` **АБО** «унікальна для платіжного застосунку надавача платіжних послуг послідовність символів, яка містить **назву домену надавача платіжних послуг**».

### Інтерпретація для Finly

**Finly не є надавачем платіжних послуг (НПП).** За Законом «Про платіжні послуги» НПП — це банки, небанківські фінустанови, оператори платіжних систем. Finly — це SaaS-обгортка над генератором QR; ми не маємо ліцензії НПП і не плануємо її здобувати в MVP.

Отже:

- **Для 002** наш домен `pay.finly.com.ua/qr/` **не дозволений нормативом**. Дозволено лише `qr.bank.gov.ua` або `bank.gov.ua/qr`.
- **Для 003** «персоніфікований код старту застосунку» **також не дозволений** нашому домену — він призначений для НПП (наприклад, `https://send.monobank.ua/qr/` для Monobank, `https://privat24.ua/qr/` для PrivatBank). Finly не може законно вписати свій домен у це поле, бо «персоніфікований» означає **домен надавача платіжних послуг**, а не довільної SaaS-обгортки.

### Висновок: `mandates: bank.gov.ua`

Норматив **жорстко** обмежує дозволені значення поля «Код старту застосунку». Для нашого випадку (Finly як SaaS, не НПП) єдині нормативно-правильні варіанти:

- `https://qr.bank.gov.ua/` — рекомендований для 003 (NEW).
- `https://bank.gov.ua/qr/` — legacy-варіант 002.

**Рішення для архітектури конфігурації:**

| Версія | Дозволений host                                                                                                                                                                          | Як зберігається у Finly                                                                              |
| ------ | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------- |
| 002    | `bank.gov.ua/qr/` (з двох нормативно-дозволених — обираємо легасі-сумісний; `qr.bank.gov.ua` для 002 ніколи не використовується, щоб не плутати парсери, що очікують саме legacy-prefix) | **Hardcoded константа** `URL_PREFIX_002` у `packages/types/src/qr/url-prefix.ts`                     |
| 003    | `qr.bank.gov.ua` (`NBU_HOST_PRIMARY`) **і** `bank.gov.ua/qr` (`NBU_HOST_LEGACY`) — **обидва одночасно**                                                                                  | **Дві named-константи** у `packages/types/src/qr/url-prefix.ts`; жодного env, жодного дефолту в коді |

**Чому 002 hardcoded.** Норматив фіксує фактично один реалістичний варіант для 002 (legacy host). Якщо коли-небудь припиниться підтримка 002 в банках — сам формат буде вилучений з `PAYLOAD_VERSIONS`, не змінений host. "Дозвіл" обирати host для 002 — фантомна гнучкість.

**Чому 003 — обидва host-и одночасно (Sprint 3 рішення A2).** Норматив дає реальний вибір між двома host-варіантами для 003, і апріорі невідомо, який з них реальніше працює у всіх 11 банк-додатках. Замість того, щоб блокувати запуск UAT-метрикою (QR-6 у попередній моделі) і обирати один host — публічна сторінка показує **дві кнопки** «Інший банк» (на `NBU_HOST_PRIMARY`) і «Інший банк (запасний варіант)» (на `NBU_HOST_LEGACY`), плюс **дві QR-картинки** на ті самі адреси. Клієнт сам пробує — якщо одна не спрацювала, тапає другу. Реальні дані по натисканнях за 2+ тижні після запуску (post-launch metric) дозволяють вирішити, чи прибрати запасну кнопку — див. QR-6 у [`docs/manual-checks/README.md`](../../manual-checks/README.md).

**За що відповідає валідація.** Whitelist двох допустимих значень — `ALLOWED_NBU_PAYLOAD_LINK_HOSTS_003` у `packages/types/src/qr/url-prefix.ts`. Перевірка відбувається у `buildNbuPayloadLink` — захист від inline-літералів у callsite, що передасть довільний рядок замість однієї з двох констант. Caller (`QrService.renderForNbuPayload`) приймає host required-параметром (TypeScript-overload блокує виклик без host для версії '003'); env-layer **знято** разом зі змінною `NBU_PAYLOAD_LINK_HOST` — обидва host-и compile-time константи, fail-fast інваріант на старті API більше не потрібен.

### Host вирішення (post-launch metric)

**Статус:** post-launch — виконується **через 2+ тижні після запуску публічної сторінки**, не блокує launch.

Sprint 3 рішення A2 (`docs/sprints/03-cabinet-public/planning-questions.md`) знімає QR-6 з блокаторів запуску. Замість матриці «11 банків × 2 ОС × 2 host-и» до launch-у — реальна аналітика на public-сторінці після launch-у:

- скільки разів натиснули «Інший банк» (primary host),
- скільки разів натиснули «Інший банк (запасний варіант)» (legacy host),
- з кожної когорти — скільки завершилось реальною оплатою.

**Правило прийняття рішення:**

| Сценарій (post-launch, 2+ тижні даних)              | Дія                                                                  |
| --------------------------------------------------- | -------------------------------------------------------------------- |
| Primary дає 99%+ натискань і таку ж частку оплат    | Прибрати legacy-кнопку (зайвий вибір без користі)                    |
| Legacy дає помітну частку (5%+) натискань або оплат | Залишити обидві — legacy рятує клієнтів, чий банк не приймає primary |
| Натискань мало (1-2 за два тижні)                   | Чекаємо ще місяць — рано вирішувати                                  |
| Натискань багато, оплат мало                        | Зламано не host — потрібна окрема діагностика                        |

Format 002 host — `bank.gov.ua/qr/` завжди (hardcoded constant, поза цим рішенням).

## Послідовність генерації (нормативна процедура)

### Формат 002 (§IV.14, ст. 21)

1. Заповнити елементи структури за таблицею 2.
2. Закодувати конкатенацію `Lf`-joined полів алгоритмом Base64URL.
3. Скласти гіперпосилання: `КодСтартуЗастосунку` + результат пункту 2 (без розділювача між ними).
4. Згенерувати графічний QR з рівнем корекції `M` або `Q` (за §III.6 ст. 4 + §III.13 ст. 5; `L` заборонено через лого).
5. Накласти знак гривні у фоновому колі (обовʼязково для 002).

### Формат 003 (§IV.10, ст. 28)

Ідентична послідовність, окрім розділювача (виключно `Lf`) і поля 4 «Функція» (`UCT`/`ICT`/`XCT`).

### Декодування (§IV.15, ст. 22 для 002; §IV.11, ст. 28 для 003)

1. Розкодувати графічний QR у гіперпосилання.
2. Виключити з гіперпосилання «Код старту застосунку» (для 002 — фіксовано 23 B; для 003 — змінна довжина, відрізається до першого символу не-`A`-кодування або за списком відомих префіксів).
3. Декодувати залишок Base64URL → отримати закодовану структуру.

## Кодування `*`-полів — нормативна асиметрія read/write

§II.4.3, ст. 26 для 003 (та аналогічно §II.4.10, ст. 19 для 002):

> «Для версії формату QR-коду 003 у полі "Кодування" значення повинно бути "2" або "1", що передбачає використання кодування Win1251 або UTF-8 відповідно для всіх полів, які можуть містити кириличні символи».

**Висновок для builder-а:** для нових payload-ів використовуємо **значення `1` (UTF-8)** — це консистентно зі всім стеком Finly (TS string = UTF-16, JSON = UTF-8, Mongo = UTF-8). Win1251 тримається для **read-side compatibility** з QR, які ми генеруватимемо та які можуть прийти від legacy-систем; для нашого write-path UTF-8 єдиний.

Парсер декодера (Sprint 2 §2.3 round-trip тест через `jsqr`) має вміти читати **обидва** значення (1 і 2), бо реальні банк-приклади з PDF використовують `2` (Win1251) — див. таблиця 3 ст. 30 для 003 і таблиці 3, 5, 7 для 002. Це не блокує MVP — ми генеруємо тільки UTF-8 і round-trip-тестуємо тільки UTF-8.

## Шпаргалка для імплементації

Коли `packages/types/src/qr/limits.ts` створюється у §2.1, ось консолідовані числа з PDF (з посиланнями для перевірки):

```ts
// see qr-spec/README.md "Формат 002" / "Формат 003"
export const FIELD_LIMITS = {
    '002': {
        receiverName: { chars: 140, bytes: 280 }, // п. 6, спека 140 C; bytes ~ chars*2 для cyrillic UTF-8
        iban: { chars: 29, bytes: 29 }, // п. 7, fixed 29 B
        amountCurrency: { chars: 15, bytes: 15 }, // п. 8, 15 B (ASCII-only: UAH...)
        receiverTaxId: { chars: 10, bytes: 10 }, // п. 9, 10 B (ASCII цифри)
        purpose: { chars: 420, bytes: 840 }, // п. 12, 420 C
        // payload-overall:  ≤ 507 B (§IV.11)
    },
    '003': {
        receiverName: { chars: 140, bytes: 280 }, // п. 6
        iban: { chars: 29, bytes: 29 }, // п. 7
        amountCurrency: { chars: 15, bytes: 15 }, // п. 8
        receiverTaxId: { chars: 10, bytes: 10 }, // п. 9
        categoryPurpose: { chars: 9, bytes: 9 }, // п. 10, формат CCCC/PPPP
        reference: { chars: 35, bytes: 35 }, // п. 11
        purpose: { chars: 420, bytes: 840 }, // п. 12
        display: { chars: 140, bytes: 280 }, // п. 13
        fieldLockMask: { chars: 4, bytes: 4 }, // п. 14, hex 0..FFFF
        validUntil: { chars: 14, bytes: 14 }, // п. 15, РРММДДГГХХСС
        issuedAt: { chars: 14, bytes: 14 }, // п. 16
        signature: { chars: 90, bytes: 90 }, // п. 17, RFU
        // payload-overall:  ≤ 507 B (§IV.8)
    },
} as const;
```

**Bytes для C-полів — оцінка не норматив.** Норматив дає тільки `C` (chars). Bytes-стелю `chars * 2` ставимо як консервативну оцінку для cyrillic UTF-8 (більшість cyrillic кодпоїнтів — 2 B; базовий ASCII — 1 B; апостроф `’` U+2019 — 3 B; емоджі — 4 B). При перевищенні `bytes` builder падає на per-field assertion, а не на сумарному 507-B обмеженні — це даж краща UX-помилка, ніж "загальний overflow".

```ts
export const FIELD_ORDER = {
    '002': [
        'BCD', // 1
        '002', // 2
        encoding, // 3
        'UCT', // 4
        '', // 5  BIC, RFU
        name, // 6
        iban, // 7
        amountCurrency, // 8 ('UAH<n>' або '')
        taxId, // 9
        '', // 10 Ціль, RFU
        '', // 11 Reference, RFU
        purpose, // 12
        '', // 13 Відображення, RFU
    ],
    '003': [
        'BCD', // 1
        '003', // 2
        encoding, // 3
        function_, // 4 (UCT/ICT/XCT)
        '', // 5  Унікальний ID, RFU
        name, // 6
        iban, // 7
        amountCurrency, // 8
        taxId, // 9
        categoryPurpose, // 10 (CCCC/PPPP)
        reference, // 11 (опц.)
        purpose, // 12
        display, // 13 (опц.)
        fieldLockMask, // 14 (опц.)
        validUntil, // 15 (опц.)
        issuedAt, // 16 (опц.)
        '', // 17 Підпис, RFU
    ],
} as const;
```

**Trailing-empty критично.** Кожна версія має точну кількість елементів масиву (13 для 002, 17 для 003). Builder mappить input → array цієї довжини, навіть якщо хвостові поля порожні. Це гарантовано тестом "кількість `Lf` у payload = N-1 для версії N полів" (§2.1).

URL-prefix константи вже існують у `packages/types/src/qr/url-prefix.ts` (з кінця Етапу A) і експортуються через **root** (`@finly/types`), бо subpath-export `@finly/types/qr` потребує moduleResolution-міграції API на `node16`/`bundler` (несумісно з існуючими CJS namespace-imports у `apps/api`, наприклад `cookie-parser`). Перехід на subpath — окремий prerequisite-крок для §2.1, де він стає виправданим через зростаючу кількість файлів у `qr/`. Зараз з root-export-у достатньо — імпортуємо `from '@finly/types'`:

```ts
// 002: hardcoded по нормативу (§I таблиця 1, ст. 15) — fixed 23 B prefix.
export const URL_PREFIX_002 = 'https://bank.gov.ua/qr/' as const;

// 003: дві named-константи (Sprint 3 рішення A2) — обидва host-и одночасно

export const NBU_HOST_PRIMARY = 'qr.bank.gov.ua' as const;
export const NBU_HOST_LEGACY = 'bank.gov.ua/qr' as const;

export const ALLOWED_NBU_PAYLOAD_LINK_HOSTS_003 = [
    NBU_HOST_PRIMARY,
    NBU_HOST_LEGACY,
] as const;

// Whitelist валідує buildNbuPayloadLink — захист від inline-літералів у
// callsite. На API-боці:
//   buildNbuPayloadLink('003', base64url, { host: NBU_HOST_PRIMARY })
//   buildNbuPayloadLink('003', base64url, { host: NBU_HOST_LEGACY })
```
