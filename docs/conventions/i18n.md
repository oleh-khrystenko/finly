# i18n — Language Synchronization Convention

> Правило описує як мова користувача синхронізується між фронтендом та бекендом.

## Принцип

```
Frontend = мова користувача (uk/en)
Backend  = тільки англійська (завжди)
Emails   = мова користувача
```

## Backend: тільки англійська

### API responses

Бекенд **НІКОЛИ** не повертає user-facing тексти. Відповідь складається з двох частин:

1. **code** — машиночитаний код для фронтенду (маппиться на локалізоване повідомлення)
2. **message** — коротке пояснення англійською для розробника (логи, дебаг)

#### Success response

```typescript
// Контролер повертає:
return {
    data: {
        code: 'MAGIC_LINK_SENT',   // фронтенд маппить на i18n
        message: 'Magic link sent', // для розробника
    },
};
```

#### Error response (через AllExceptionsFilter)

```json
{
    "error": {
        "code": "RATE_LIMIT_EXCEEDED",
        "message": "Too many requests. Try again in 15 minutes."
    }
}
```

- `code` — з enum `ERROR_CODE` або кастомний success/info код
- `message` — англійською, лаконічно, для розробника

### Що заборонено на бекенді

- Повертати текст українською або будь-якою іншою мовою
- Повертати user-facing повідомлення (тости, нотифікації)
- Хардкодити локалізований текст у responses
- Використовувати i18n бібліотеки на бекенді для API responses

### Emails — виняток

Emails — єдине місце де бекенд працює з локалізованим контентом.
Email templates мають підтримувати мову користувача (`user.preferredLang`).

```typescript
// EmailService приймає locale
async sendMagicLink(email: string, token: string, lang: Lang): Promise<void>
```

## Frontend: мова користувача

### Маппинг кодів на повідомлення

Фронтенд отримує `code` з API та маппить на локалізоване повідомлення через `messages/{locale}.json`.

```
API code: "MAGIC_LINK_SENT"
  → i18n key: "notifications.auth.magic_link_sent"
  → uk: "Посилання надіслано на вашу пошту"
  → en: "Link sent to your email"
```

### i18n ключі для API кодів

Паттерн для маппингу API кодів:

```
notifications.{module}.{code_snake_case}  — success/info повідомлення
errors.{module}.{code_snake_case}         — error повідомлення
errors.generic.{code_snake_case}          — загальні помилки (fallback)
```

### Fallback стратегія

Якщо для `code` немає відповідного i18n ключа — використовувати `errors.generic.unknown`:
- UK: "Сталася помилка. Спробуйте пізніше"
- EN: "An error occurred. Please try again later"

**НІКОЛИ** не показувати `message` з API користувачу — це текст для розробника.

## Response Code Registry

Всі коди (success, error та будь-які майбутні) зареєстровані в єдиному enum `RESPONSE_CODE` (`packages/types/src/enums/response-code.ts`).

Кожен код має **тип** (`ResponseType`) — визначає як фронт його відображає:

| ResponseType | Призначення | UI |
|-------------|-------------|----|
| `success` | Операція виконана | зелений |
| `error` | Помилка | червоний |

Нові типи (наприклад `warning`, `info`) додаються коли з'явиться потреба.

### Як додати новий код

1. Додай рядок в `RESPONSE_CODE`
2. Додай маппінг в `RESPONSE_CODE_TYPE` (код → тип)
3. Додай i18n ключ в `messages/uk.json` та `messages/en.json`

## Синхронізація мови

### Звідки бекенд знає мову користувача?

1. **Авторизований запит** — з `user.preferredLang` (MongoDB)
2. **Неавторизований запит** (наприклад, magic-link/send) — фронт передає `lang` у body або header
3. **Рекомендований підхід**: header `Accept-Language` для неавторизованих запитів

### Оновлення мови

Коли користувач змінює мову на фронті — фронт надсилає запит на API для збереження `preferredLang`.
