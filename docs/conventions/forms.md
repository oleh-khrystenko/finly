# Form Validation Feedback

> Кнопка ніколи не блокується, щоб просигналізувати провал валідації. Вона
> лишається клікабельною, а причина показується простою, зрозумілою користувачу
> мовою.

## Принцип

Заблокована кнопка без пояснення читається як «сайт зламався»: користувач не
розуміє, чому клік не працює, і що від нього хочуть. Правильна поведінка —
дозволити клік і показати причину, щоб людина прийняла свідому дію.

| Поведінка                                                          | Оцінка       |
| ------------------------------------------------------------------ | ------------ |
| Валідація не пройшла → кнопка `disabled`, причини не видно          | ❌ заборонено |
| Валідація не пройшла → клік дозволено, причина видима простою мовою | ✅ правильно  |

## Rules

1. **Кнопка сабміту/дії блокується ЛИШЕ на `loading` / `submitting`** (захист від
   подвійного сабміту), **ніколи на валідності форми чи вмісті поля**. У
   `UiButton` `loading` вже блокує клік (`blocked = disabled || loading`), тож
   окремий `disabled` для цього не потрібен.

2. **На невалідному вводі показуй причину, а не гаси кнопку:**
    - field-level помилка → під полем через `error`-проп (`UiInput`, `UiTextarea`,
      `UiPasswordInput`, `UiSelect`);
    - form-level блокер (стан, не привʼязаний до одного поля) → `toast` або
      inline-повідомлення біля кнопки.

3. **Причина — нетехнічною мовою** (див. [tone.md](tone.md)). Не машинний код і не
   жаргон: «Сума не може бути відʼємною», а не «INVALID_AMOUNT_NEGATIVE»; «Питання
   задовге. Скоротіть його до 4000 символів», а не мовчазний over-limit.

4. **React Hook Form**: не пиши `disabled={!formState.isValid}`,
   `disabled={!isDirty}`, `disabled={!fieldValue}`. Дозволяй сабміт — resolver
   сам покаже помилки під полями. Захисний save-guard (early-return / no-op при
   невалідному стані) лишається, але **кнопка клікабельна** — guard ловить race,
   а користувач бачить причину, не мертву кнопку.

5. **Порожній ввід** (ще нічого не введено) можна лишити `disabled` для чистого
   «нема що надсилати» (порожній чат-інпут), бо це самоочевидний стан, а не провал
   валідації. Але щоймить зʼявляється вміст, який не проходить перевірку —
   застосовується правило 2.

## Виняток: `disabled` за відсутності дії

`disabled` легітимний, коли дії **обʼєктивно немає** — це не «валідація не
пройшла», а «робити нема чого»:

- межа впорядкування («вгору» на першому елементі, «вниз» на останньому);
- порожній список для дії («прикріпити», коли прикріпляти нема що);
- поле, заблоковане бізнес-правилом, з поясненням поруч (напр. slug
  опублікованого гайда);
- поки триває незворотна операція (`loading` / pending).

У цих випадках причина або самоочевидна (сіра стрілка «вгору» на верхньому
рядку), або підписана поруч.

## Приклади

| Контекст                          | Correct                                                      | Wrong                                     |
| --------------------------------- | ----------------------------------------------------------- | ----------------------------------------- |
| Невалідна дата (inline-edit)      | Клік → «Введіть дату у форматі ДД.ММ.РРРР» під полем         | Кнопка «Зберегти» сіра, причини не видно  |
| Невалідна сума                    | Клік → «Сума не може бути відʼємною» під полем               | Кнопка `disabled={parseErr !== null}`     |
| Питання помічнику довше за ліміт   | Клік → «Питання задовге. Скоротіть його до N символів»       | Кнопка `disabled={overLimit}`, лише лічильник |
| Порожнє поле пароля перед видаленням | Клік → «Введіть пароль для підтвердження» під полем        | Кнопка `disabled={!password}`             |
| «Вгору» на першому елементі списку | Кнопка `disabled` (дії немає, стан самоочевидний)           | —                                         |
